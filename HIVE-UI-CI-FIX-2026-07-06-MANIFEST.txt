1. src/test/LoginScreen.test.tsx
---------------------------------------------------------------------
Root cause:
  import.meta.env.DEV defaults to TRUE under Vitest (Vite computes
  DEV = mode !== 'production'; Vitest's default test mode is "test",
  not "production"). The test file's own comment claiming DEV was
  false by default was incorrect.

  Because DEV was actually true, AuthContext.tsx's local dev-session
  shortcut fired instead of the real-backend branch these tests are
  built to exercise:
    - restoreSession() skipped its fetch entirely.
    - login() skipped loginUi() and went straight to refreshHealth(),
      which became the first real fetch call and consumed the mock
      meant for the session check — surfacing a stray "no session"
      error even on a successful login, and letting the submit button
      re-enable before the "disabled while in flight" test could
      observe it.

Fix:
  Added `vi.stubEnv('DEV', false)` in beforeEach and `vi.unstubAllEnvs()`
  in afterEach, forcing AuthProvider onto the real-backend branch the
  fetch mocks are written for. Corrected the misleading comment.

Blast radius:
  AuthContext.tsx is the only source file that reads DEV/PROD anywhere
  in the codebase, and this test file is the only test that touches
  those flags — no other tests or app behavior affected.

---------------------------------------------------------------------
2. vitest.config.ts
---------------------------------------------------------------------
Root cause:
  No include/exclude scoping meant Vitest's default glob matched every
  *.test.*/*.spec.* file in the repo, not just src/test/. Two file
  types got swept in by accident:
    - scripts/security.test.mjs and scripts/ux-contract.test.mjs use
      Node's native `node:test` runner (security.test.mjs already has
      its own script, `test:security`; ux-contract.test.mjs had none).
      Running them under Vitest broke import.meta.url resolution,
      producing spurious ENOENT errors for source files that in fact
      exist (verified below) — the files were never missing, only
      misexecuted.
    - e2e/critical-path.spec.ts is a Playwright spec (testDir: './e2e'
      in playwright.config.ts, run via `npm run test:e2e`). Playwright's
      test.describe() throws when invoked outside the Playwright
      runner, which is exactly the second error seen in CI.

Fix:
  Added explicit `include: ['src/test/**/*.{test,spec}.{ts,tsx}']` and
  `exclude: ['e2e/**', 'scripts/**', 'node_modules/**']` so Vitest only
  picks up the unit tests it actually owns.

Verified (see manifest below) that ConfirmDialog.tsx, ModelPicker.tsx,
and ChatPage.tsx all exist and their content matches every assertion
in scripts/ux-contract.test.mjs — confirmed by grepping each exact
pattern the test checks for. Nothing was actually broken in app code.

3. package.json
---------------------------------------------------------------------
Follow-on from the vitest.config.ts fix: excluding
scripts/ux-contract.test.mjs from Vitest without wiring it into its
own script would mean a real, passing UX-regression test silently
stops running in CI altogether (it was previously only "running" by
accident, and failing, via Vitest's broad glob).

Fix:
  - Added "test:ux-contract": "node --test scripts/ux-contract.test.mjs"
  - Added it to the `check` pipeline, alongside test:security:
    verify:lock -> verify:source -> test:security -> test:ux-contract
    -> typecheck -> lint -> test:unit -> build -> verify:dist
